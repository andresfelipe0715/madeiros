# Report: Order Stage Queue Logic & Interruption Handling

I have analyzed the system's logic regarding how orders are queued and processed across production stages. Specifically, I investigated what happens when a "lower stage" (previous stage) sends an order that has an ID lower than the one currently being worked on.

## Summary of Findings

The system currently enforces a **strict FIFO (First-In, First-Out) queue based on Order ID**. This enforcement has a significant impact on workers when orders arrive out of sequence.

### 1. Behavior when a HIGHER ID Order arrives
If a worker is processing **Order #10** and **Order #11** becomes ready for the same stage:
*   **No change**: The worker can continue and finish Order #10 without any interruption.
*   The system correctly recognizes that Order #10 is older (and thus higher priority) than Order #11.

### 2. Behavior when a LOWER ID Order arrives
If a worker is processing **Order #10** and **Order #5** suddenly becomes ready (e.g., it was delayed in a previous stage or was "remitted" back):
*   **Action Blocked**: The worker will be **instantly blocked from finishing** Order #10.
*   **UI Feedback**: The "Finalizar" (Finish) button for Order #10 will turn gray (disabled) and show a tooltip: *"Este pedido no es el siguiente en la fila."*
*   **Queue Status**: Order #5 will appear in the table with a **"Siguiente" (Next)** badge, indicating it must be processed before any order with a higher ID.

## Technical Reason
This behavior is rooted in the `StageAuthorizationService::isNextInQueue` logic. This function explicitly checks if **any** order for the same stage exists that:
1. Has a **lower ID** than the current one.
2. Is **not completed**.
3. Is **not marked as "Pendiente"** (Blocked by Admin).
4. Has its **previous stages completed**.

Crucially, **the check ignores whether the current order is already "In Process" (`started_at`)**.

## Implications for the User
*   **Strict Integrity**: The system ensures that the oldest orders are always prioritized, preventing newer orders from "jumping the line" completely.
*   **Worker Interruption**: A worker who has already invested time into an order might find themselves unable to finish it if an older order arrives at their station mid-work.
*   **Administrator Override**: Only users with **Admin permissions** (can edit orders) can bypass this queue rule. They will see the button enabled even if the order is not next in line.
*   **Workaround**: If a worker is blocked by a newly arrived older order, they can use the **"Detener proceso" (Pause)** button on their current order, start and finish the older one, and then resume their previous work.
