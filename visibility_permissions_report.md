# Role Visibility Permissions Report

This report summarizes the structure and purpose of the visibility permission system within the application.

## Core Table: `role_visibility_permissions`

The visibility system is controlled by the `role_visibility_permissions` table, which maps specific UI/Data access triggers to user roles.

| Column | Purpose |
| :--- | :--- |
| `can_view_files` | Master switch for the **Files** section in the order view. If false, the entire section is hidden. |
| `can_view_order_file` | Controls access to the **Order/Invoice PDF** (the main document uploaded during creation). |
| `can_view_machine_file` | Controls access to **Technical/CNC files** intended for operators. |

## Implementation Details

### Fail-Safe Logic
As implemented in `VisibilityService.php`, the system defaults to **TRUE** if no permission record exists for a role. This ensures that users are not locked out of critical files if the database is missing a specific configuration.

### Discovered "Phantom" Permissions
During the investigation, several columns were found in the `VisibilityPermissionSeeder` that **do not exist** in the database:
- `can_view_notes`
- `can_view_remit_history`
- `can_view_pending_reason`

These have been removed from the seeder to prevent SQL errors and maintain schema integrity.

## Seeder Updates
The `DefaultDataSeeder` has been updated to automatically initialize these permissions for ALL roles:
- **Admin**: Full access (All True).
- **Other Roles**: Defaulted to Full access (All True) unless manually changed in the DB.
